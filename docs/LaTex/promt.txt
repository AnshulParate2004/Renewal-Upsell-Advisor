MISSION STATEMENT
-----------------
Build an enterprise-grade, autonomous AI agent that proactively monitors B2B SaaS customer health 24/7, predicts churn risks and upsell opportunities, and AUTONOMOUSLY EXECUTES revenue-protecting workflows including:
- Automated quote generation with embedded payment links
- Closed-loop voice outreach with intelligent retry logic

The goal is to maximize Net Revenue Retention (NRR) by reducing churn 20-30% through intelligent, timely interventions WITHOUT requiring human CSM involvement.

1.1 TECHNOLOGY STACK
--------------------
Backend Framework: FastAPI (Python 3.11+)
Primary Database: PostgreSQL (Structured data - Accounts, Contracts, Opportunities)
Document Store: MongoDB (Unstructured logs - Voice transcripts, Email bodies, Webhooks)

Payment Processing: Stripe (Embedded payment links in quotes)
CRM Integration: Salesforce (Bi-directional sync)
ML/AI Stack: 
  - Sentiment Analysis: HuggingFace Transformers (distilbert-base-uncased-finetuned-sst-2-english)
  - Churn Prediction: XGBoost or LightGBM
  - Upsell Detection: Collaborative Filtering + Rule-based triggers
Containerization: Docker + Docker Compose
API Documentation: OpenAPI/Swagger (auto-generated by FastAPI)

1.3 DATABASE SCHEMA DESIGN
---------------------------

PostgreSQL Tables:

1. accounts
   - id (UUID, PK)

   - id (UUID, PK)
   - account_id (UUID, FK -> accounts.id)
   - start_date (DATE)

PART 2: THE 24-HOUR ANALYSIS ENGINE
================================================================================

2.1 SCHEDULER CONFIGURATION
----------------------------
Trigger: Daily at 00:00 UTC (using APScheduler or Celery Beat)
Execution Time: ~15-30 minutes for 10,000 accounts
Parallelization: Process accounts in batches of 100 using asyncio

2.2 DATA COLLECTION PIPELINE
-----------------------------
For each account, fetch the LAST 24 HOURS of:
1. Support Tickets (from Salesforce or Zendesk)
2. Email Interactions (from MongoDB event_logs)
3. Product Usage Logs (API calls, feature usage, login frequency)
4. Voice Call Transcripts (from MongoDB voice_transcripts)

2.3 AI MODEL EXECUTION (3 Concurrent Models)
---------------------------------------------

MODEL 1: Sentiment Analyzer
- Input: Concatenated text from emails + support tickets + voice transcripts
- Model: HuggingFace distilbert-base-uncased-finetuned-sst-2-english
- Output: Sentiment score (-1.0 to 1.0) and label ('positive', 'neutral', 'negative')
- Update: accounts.sentiment field in PostgreSQL
- Logic:
  * Score > 0.3 → 'positive'
  * Score -0.3 to 0.3 → 'neutral'
  * Score < -0.3 → 'negative'

MODEL 2: Churn Predictor
- Input Features:
  * License utilization (%)
  * Support ticket volume (last 7 days)
  * Sentiment score (from Model 1)
  * Days since last login
  * Feature adoption rate
  * Payment history (late payments)
- Model: XGBoost binary classifier (trained on historical churn data)
- Output: Churn probability (0-100%)
- Update: accounts.risk_score field in PostgreSQL
- Threshold: risk_score > 70 → Trigger high-priority intervention

MODEL 3: Upsell Detector
- Input Features:
  * License utilization (%)
  * Feature usage patterns
  * Account growth trajectory (user count over time)
  * Industry benchmarks
- Logic (Rule-based + ML):
  * IF license_utilization > 85% → Flag for seat expansion
  * IF using advanced features at high frequency → Flag for tier upgrade
  * ML Model: Collaborative filtering to predict next likely purchase
- Output: Upsell propensity score (0-100%)
- Update: accounts.upsell_propensity field in PostgreSQL
- Action: IF upsell_propensity > 60 → Auto-create opportunity record

2.4 OUTPUT & ACTIONS
--------------------
After analysis, the system:
1. Updates PostgreSQL fields: risk_score, sentiment, upsell_propensity
2. Logs analysis results to MongoDB event_logs
3. Triggers downstream workflows:
   - IF risk_score > 70 AND contract.end_date within 90 days → Trigger early renewal workflow
   - IF upsell_propensity > 60 → Create opportunity and generate quote

================================================================================
PART 3: YAGNA-STYLE NOTIFICATION WORKFLOWS
================================================================================

3.1 TIME-BASED TRIGGERS
------------------------
The system checks contract.end_date daily and triggers workflows at:

T-90 DAYS (Early Warning)
- Action: Generate renewal quote PDF
- Email: "Your contract renews in 90 days. Here's your early renewal offer."
- Attachment: Quote PDF with 5% early renewal discount
- Update: contracts.renewal_stage = 't_90'

T-60 DAYS (Adoption Check)
- Action: Analyze product usage and adoption metrics
- Logic:
  * IF license_utilization < 50% → Email: "We noticed low usage. Let's schedule a training session."
  * IF license_utilization > 85% → Email: "You're maxing out your licenses. Consider upgrading."
- Update: contracts.renewal_stage = 't_60'

T-30 DAYS (Urgent Renewal)
- Action: Generate final renewal quote with embedded Stripe payment link
- Email: "Your contract expires in 30 days. Renew now with one click."
- Payment Link: Embedded Stripe checkout link (auto-generated via Stripe API)
- Update: contracts.renewal_stage = 't_30'
- Escalation: IF risk_score > 70 → ALSO trigger voice call workflow

3.2 QUOTE GENERATION LOGIC
---------------------------
File: app/services/workflows/quote_generator.py

Function: generate_renewal_quote(contract_id: UUID) -> str
Steps:
1. Fetch contract details from PostgreSQL
2. Calculate renewal value:
   - Base value = contract.value
   - IF T-90 → Apply 5% early renewal discount
   - IF upsell_propensity > 60 → Add recommended seat expansion
3. Generate PDF using ReportLab or WeasyPrint
4. Upload PDF to S3 or cloud storage
5. Create Stripe payment link:
   - stripe.PaymentLink.create(
       line_items=[{price: renewal_price_id, quantity: 1}],
       after_completion={type: 'redirect', redirect: {url: success_url}}
     )
6. Store quote_pdf_url and stripe_payment_link in opportunities table
7. Return quote_pdf_url

3.3 EMAIL DISPATCH
------------------
File: app/services/workflows/email_sender.py

Function: send_renewal_email(account_id: UUID, stage: str)
Steps:
1. Fetch account and contract details
2. Select email template based on stage (t_90, t_60, t_30)
3. Personalize email with:
   - Account name
   - Contract end date
   - Quote PDF link
   - Stripe payment link (for T-30 only)
4. Send via SendGrid or AWS SES
5. Log email event to MongoDB event_logs

================================================================================
PART 4: CLOSED-LOOP VOICE AGENT
================================================================================

4.1 VOICE CALL INITIATION
--------------------------
File: app/services/voice_agent/call_handler.py

Function: initiate_call(account_id: UUID, reason: str) -> dict
Trigger Conditions:
- risk_score > 70 AND contract.end_date within 30 days
- Opportunity value > $50,000 (high-value accounts)
- Manual trigger from CSM dashboard

Steps:
1. Fetch account contact phone number from PostgreSQL
2. Create Twilio call:
   client.calls.create(
     to=contact_phone,
     from_=twilio_number,
     url=f"{base_url}/api/v1/webhooks/twilio/voice",
     status_callback=f"{base_url}/api/v1/webhooks/twilio/status"
   )
3. Create interaction record in PostgreSQL:
   - type = 'voice_call'
   - outcome = 'pending'
   - retry_count = 0
4. Return call_sid

4.2 CALL OUTCOME PROCESSING
----------------------------
File: app/api/v1/endpoints/webhooks.py

Endpoint: POST /api/v1/webhooks/twilio/status
Twilio sends status callbacks with CallStatus:
- 'completed' → Call was picked up
- 'no-answer', 'busy', 'failed' → Call was missed

Logic:
IF CallStatus == 'completed':
  1. Update interaction.outcome = 'success'
  2. Fetch call recording and transcript from Twilio
  3. Store transcript in MongoDB voice_transcripts
  4. Run sentiment analysis on transcript
  5. Update account.sentiment based on call sentiment
  6. Log success event

ELSE (Call missed):
  1. Update interaction.outcome = 'missed'
  2. Increment interaction.retry_count
  3. IF retry_count < 3:
     - Calculate next_retry_at = current_time + 4 hours
     - Schedule retry task in Redis queue
  4. Log missed event to MongoDB

4.3 RETRY SCHEDULER
-------------------
File: app/services/voice_agent/retry_scheduler.py

Function: schedule_retry(interaction_id: UUID)
Steps:
1. Fetch interaction record
2. Calculate next_retry_at:
   - Retry 1: +4 hours
   - Retry 2: +8 hours (next business day morning)
   - Retry 3: +24 hours
3. Add task to Redis queue with delay
4. Update interaction.next_retry_at in PostgreSQL

Background Worker:
- Celery worker polls Redis queue
- When retry time arrives, calls initiate_call() again
- Updates retry_count
